

# Аудит: Безопасность, Механики и Функции игры

## Критические уязвимости (требуют исправления)

### 1. Продажа предметов без серверной валидации
**Файл:** `src/components/game/improved/InventoryManager.tsx:97-112`

Продажа предмета выполняется **полностью на клиенте**: удаляется item instance, затем вызывается `addBalance(sellPrice)` в Zustand, который через `useGameSync` записывает новый баланс в БД. Злоумышленник может:
- Перехватить `addBalance` и вызвать с произвольной суммой
- Модифицировать `sell_price` в памяти перед продажей
- Вызывать `addBalance` напрямую из консоли

**Исправление:** Создать Edge Function `sell-item` которая:
1. Принимает `item_instance_id` и `wallet_address`
2. Проверяет владение предметом
3. Получает `sell_price` из `item_templates` на сервере
4. Атомарно удаляет предмет + начисляет баланс через `atomic_balance_update`

### 2. Опыт аккаунта рассчитывается на клиенте
**Файл:** `src/stores/gameStore.ts:74-90`, `src/components/game/adventures/AdventuresTab.tsx:67`

`addAccountExperience` в Zustand позволяет начислить любой опыт, а `useGameSync` запишет его в БД. Игрок может повышать уровень аккаунта вызовом из консоли.

**Исправление:** Перенести начисление опыта в `apply_battle_rewards` RPC (уже частично сделано для подземелий — `experience_reward`), но для режима Приключений (`AdventuresTab`) опыт начисляется клиентом.

### 3. `open-elleonor-box` принимает `wallet_address` из тела запроса
**Файл:** `supabase/functions/open-elleonor-box/index.ts:12,76`

В отличие от других функций, здесь `wallet_address` берется напрямую из request body без JWT-верификации. Кто угодно может открыть коробку от имени другого игрока или подставить чужой кошелек.

**Исправление:** Добавить ту же цепочку верификации (JWT → wallet_identities → game_data), как в `start-dungeon-session`.

### 4. Рассогласование формулы кубика между PvP и подземельями
**Файл:** `src/utils/diceFormula.ts` vs `supabase/functions/pvp-submit-move/index.ts:56-80`

| Бросок | Клиент (diceFormula.ts) | Сервер PvP |
|--------|------------------------|------------|
| 1 | 0% (контратака) | 0% (контратака) |
| **2** | **25% (скользящий)** | **0% (промах)** |
| 3 | 50% | 50% |
| 4 | 100% | 100% |
| 5 | 150% | 150% |
| 6 | 200% | 200% |

Бросок 2 дает разный результат. Pity-система существует только на клиенте.

**Исправление:** Синхронизировать серверную формулу с клиентской (25% для броска 2), добавить pity-систему на сервер.

### 5. Глобальный `pityCounter` в модуле — race condition
**Файл:** `src/utils/diceFormula.ts:16`

`let pityCounter = 0` — это глобальная переменная модуля. Если два боя идут одновременно (теоретически возможно при быстрой навигации), они разделяют один счетчик. Кроме того, pity не сбрасывается при начале нового боя автоматически.

**Исправление:** Сделать pity-счетчик per-session (передавать как параметр или хранить в battle state).

## Средние проблемы

### 6. `combatUtils.ts` — устаревшая формула кубика
**Файл:** `src/components/game/adventures/game/utils/combatUtils.ts`

Содержит старую формулу (другие модификаторы: 1=блок, 3=+30%, 4=парирование, 5=-30%, 6=+100%). Если этот файл используется в режиме Приключений, формулы расходятся с подземельями.

**Исправление:** Проверить использование и унифицировать с `diceFormula.ts`, либо удалить если не используется.

### 7. `useGameSync` — баланс перезаписывается из Zustand в БД
**Файл:** `src/hooks/useGameSync.ts:157-191`

Синхронизация отправляет `balance` из Zustand → `updateGameData` → БД. Это означает, что любое клиентское изменение баланса (включая `addBalance` от продажи) записывается в БД без серверной проверки. Фактически `useGameSync` является каналом для записи произвольного баланса.

**Исправление:** Исключить `balance` из синхронизации Zustand→БД. Баланс должен меняться только через серверные RPC (`atomic_balance_update`, `apply_battle_rewards`).

### 8. `open-card-packs` — дублирование данных cards
**Файл:** `supabase/functions/open-card-packs/index.ts:617-628`

Карты записываются и в `game_data.cards` (JSON массив), и в `card_instances` (таблица). Двойное хранение создает риск рассинхронизации. Если `card_instances` — основная таблица, то запись в `game_data.cards` избыточна.

**Исправление:** Убрать обновление `game_data.cards` в edge function, полностью перейти на `card_instances`.

### 9. `game-wipe` — не очищает `reward_claims`
**Файл:** `supabase/functions/game-wipe/index.ts`

Таблица `reward_claims` хранит ключи идемпотентности. После вайпа старые claim_key остаются, что может блокировать повторные награды в новых сессиях (маловероятно из-за UUID, но грязные данные).

**Исправление:** Добавить `DELETE FROM reward_claims` в wipe.

### 10. `game-wipe` — не очищает `security_audit_log`, `claim_nonces`, `shop_sessions`
Старые сессии и логи остаются после вайпа. Не критично, но `shop_sessions` и `claim_nonces` могут содержать устаревшие токены.

## Рекомендации по улучшению механик

### 11. Режим Приключений — полностью клиентский
Как отмечено в memory, награды в Adventure начисляются на клиенте. Это самая крупная незакрытая дыра — перенести бой/награды на сервер или хотя бы валидировать через Edge Function.

### 12. Ограничение максимальных наград подземелья
Текущий cap: `Math.max(5000, level * 600)` ELL. На уровне 100 это 60,000 ELL за один забег. Проверить, соответствует ли это экономическому дизайну.

### 13. Treasure Hunt — нет атомарной проверки `found_quantity`
**Файл:** `claim-battle-rewards/index.ts:181`

Проверка `treasureEvent.found_quantity < treasureEvent.total_quantity` не атомарна — два игрока могут одновременно получить последний предмет. Нужен `UPDATE ... WHERE found_quantity < total_quantity RETURNING ...`.
