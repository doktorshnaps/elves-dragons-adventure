
План: исправить привязку Telegram так, чтобы кнопка в настройках реально подключала аккаунт и больше не показывала ложную ошибку.

Что, скорее всего, сломано сейчас
1. Клиент проверяет статус через прямой `SELECT` из `game_data`, а в проекте для `game_data` уже есть жёсткие RLS-ограничения на прямое чтение. Из-за этого проверка после сохранения может падать даже если запись была обновлена.
2. RPC `save_telegram_chat_id` делает только `UPDATE` и ничего не возвращает. Если строки `game_data` для кошелька ещё нет или `wallet_address` не совпал, вызов не падает, но и `chat_id` не сохраняется.
3. Кнопка “Открыть бота” сейчас открывает просто чат бота, а не явный сценарий повторного входа/подключения Mini App. Для пользователя это неочевидно.

Что я предлагаю внедрить

1. Починить backend-часть привязки
- Обновить RPC `save_telegram_chat_id` так, чтобы он:
  - сначала гарантировал наличие строки игрока,
  - сохранял `chat_id`,
  - возвращал структурированный результат (`success`, `saved_chat_id`, `row_found/updated`).
- Добавить отдельный безопасный RPC для чтения Telegram-статуса по кошельку вместо прямого чтения `game_data` с клиента.

2. Убрать зависимость клиента от прямого доступа к `game_data`
- Переписать `getTelegramConnectionStatus()` и `saveTelegramChatId()` на новые RPC.
- Возвращать точные причины ошибки:
  - нет Telegram Mini App,
  - нет кошелька,
  - не найдена/не создана запись игрока,
  - `chat_id` сохранён,
  - привязан другой Telegram-аккаунт.

3. Переделать UX подключения в настройках
- Сохранить блок “Уведомления Telegram”, но сделать его пошаговым:
  - статус,
  - кнопка “Подключить / Переподключить”,
  - понятная подсказка, что подключение работает только внутри Telegram Mini App.
- Вместо общего текста “chat_id не сохранился” показывать точную причину.
- Если найден mismatch, показывать отдельный сценарий “Перепривязать”.

4. Исправить кнопку открытия Telegram
- Заменить простой переход в бота на корректный deep-link для повторного запуска нужного сценария через бота/Mini App.
- Если игра уже открыта в Telegram, не отправлять пользователя в тупиковый экран бота, а вести в ожидаемый flow подключения.

5. Стабилизировать авто-привязку
- Оставить автосохранение в `useGameInitialization`, но сделать его вторичным удобством.
- Основной надёжный путь — явная ручная привязка из настроек.
- Защитить автопривязку от повторных ложных ошибок, если статус нельзя проверить напрямую.

Файлы, которые затрону
- `supabase/migrations/...` — обновление RPC сохранения и RPC чтения статуса
- `src/utils/telegramNotifications.ts` — перевод логики на RPC и нормальные статусы
- `src/components/SettingsMenu.tsx` — новый UX подключения и точные сообщения
- `src/components/game/initialization/useGameInitialization.ts` — аккуратная автопривязка без ложных фейлов

Технически важно
```text
Сейчас:
Settings -> saveTelegramChatId() -> rpc(save_telegram_chat_id)
-> getTelegramConnectionStatus() -> direct SELECT game_data
-> RLS блокирует чтение -> UI считает, что привязка провалилась

После правки:
Settings -> rpc(save_telegram_chat_id_v2)
-> rpc(get_telegram_connection_status)
-> UI получает точный результат без прямого доступа к game_data
```

Ожидаемый результат
- кнопка “Подключить Telegram” перестанет падать на ложной проверке;
- `telegram_chat_id` будет реально сохраняться даже если запись игрока ещё не была подготовлена заранее;
- пользователь увидит понятный статус подключения;
- ручная привязка в меню станет основным и надёжным способом подключения уведомлений.
